# Skills 管理与使用规范 (Skills Management & Usage Workflow)

## 1. Skills 中文化 (Localization)
- **规则**: 任何新创建、安装或修改的 Skills，其 `SKILL.md` 文件内容必须翻译或编写为**中文**。
- **例外**: `name` 字段（元数据）保持英文以兼容 CLI 工具，但 `description` 和正文必须是中文。

## 2. 透明化使用 (Transparency)
- **规则**: 当我（Agent）在处理任务时，如果决定调用某个 Skill，必须在回复中**明确告知用户**。
- **格式示例**:
  > 💡 我将使用 `[skill-name]` 技能来辅助处理此任务...

## 3. 自动归类与目录管理 (Categorization & Organization)
- **规则**: 为了保持 Skills 目录整洁，所有 Skill **不应直接存放在根目录**。
- **执行动作**:
  1. **新建 Skill 时**: 我会根据 Skill 的功能属性，判断其所属领域（如 `DataAnalysis`, `Writing`, `Coding` 等），并将其创建在对应的**子文件夹**中（例如 `.gemini/skills/DataAnalysis/data-engineer/`）。
  2. **维护 Skill 时**: 如果发现现有 Skill 存放位置混乱或领域重叠，我有权**重构目录结构**，将相关 Skill 移动到更合理的分类文件夹下，确保“物以类聚”。
  3. **层级建议**: 建议保持 2 级目录结构（Category/SkillName），避免过深。

## 4. Skill 准入判断与克制原则 (Necessity Check)
- **规则**: 在建议或创建新 Skill 之前，必须进行**必要性评估**，避免过度封装。
- **判断标准**:
  - **🚫 拒绝创建**: 如果任务属于基础原子操作（如读写文件、简单的 HTTP 请求）、一次性脚本任务或无复用价值的临时逻辑。-> **直接使用代码解决**。
  - **✅ 建议创建**: 如果任务包含复杂的业务方法论、固定的长流程（Workflow）、需要复用的特定领域知识（Domain Knowledge）。
- **执行**: 当用户要求创建 Skill 时，如果我认为不必要，应**主动劝阻**并说明理由（“这个直接写个脚本跑一下更快，没必要封装”），除非用户坚持。

## 5. 自动迭代优化 (Self-Correction & Evolution)
- **触发条件**: 如果用户对同一个问题的处理结果表示**多次不满意**（例如明确表示“不对”、“还是不行”、“重试”）。
- **执行动作**:
  1. **反思**: 分析为什么现有 Skill 的指令没能解决问题。
  2. **总结**: 提炼出改进点（例如：需要在指令中增加针对这种情况的处理步骤）。
  3. **写入**: 自动调用 `skill-improver` 或直接编辑对应的 `SKILL.md` 文件，将这些“经验教训”作为新的规则或注意事项写入。
  4. **通知**: 告知用户：“根据您的反馈，我已更新了 `[skill-name]` 的规则，下次处理此类问题会更准确。”
